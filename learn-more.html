ADL xAPI Launch

Learn More

xAPI Launch is an algorithm that can be implemented to launch and initialize "xAPI Launch enabled" content.

It ensures that statements which claim to be part of an experience really came from that experience. xAPI Launch differs from other launch algorithms in a few important ways.

  • It avoids placing personally indentifying information (PII) in a query string. This helps protect the learner's identity.
  • It also avoids initiating the launch session from an HTTP POST request.
  • In order to avoid both PII in the query string and an initial POST, it generates a one time use random token that can be exchanged for the name and LRS endpoints of the learner, but can be safely transfered as part of the URL.

Supported Uses
  • Static HTML resources
  • HTML content rendered by a server
  • Desktop applications through a URL Protocol Handler
  • Mobile applications through a URL Protocol Handler
  • Disconnected experiences by manually entering a code

How it works

An administrator configures launch information

Instead of configuring the content, configuration is done at the server level.

The user chooses to launch some content
The launch server generates a launch token

unique id, "launch token", and returns it to the user

Launch information is delivered to the content

the user delivers the token, and launch service address, to the content (often as query string params http get request)

The content attempts to exchange the launch token

the content gets the token and launch service address and issues an http post request containing the launch token to the launch service address

The launch server verifies the launch token

a> token is valid, b> the session is in the unititialized state, c> the launch window has not timed out

The launch server initializes the session and returns configuration details

and responds from the post request with a> the actor, b> address to send the xapi statements, c> contextual information to be added to the statement about this launch. The launch server also sends a session cookie which must be included with all subsequent requests to the launch server about the session

The launch server proxies content xAPI communication to the LRS

The launch server recieves requests, adds context information to statements and sends them to the configured LRS, (does it proxy all other xapi requests to the lrs?)
